02

Como funciona HOJE

Modelo Direto — API para API — "Ligação telefônica"
PROTHEUS
—— POST /Incluir_API ——>
EME4
(1 servidor)
1. Protheus monta o JSON com os dados
2. Faz chamada HTTP direta ao EME4
3. Protheus AGUARDA... 2s, 5s, 10s, timeout?
4. EME4 responde { "Sucesso": true }
Se EME4 estiver fora → ERRO! Dados podem se perder

Problemas

  • Protheus fica travado esperando
  • Se EME4 cai, tudo para
  • Sem retry automático
  • Servidor único = ponto único de falha
  • Sem distribuição de carga

Exemplo: 500 OPs

  • Envia uma por uma, esperando cada resposta
  • Se a 300ª falhar, as 200 restantes param
  • EME4 pode sobrecarregar
  • Lento e frágil

  • Protheus não trava mais
  • Nenhum dado se perde
  • Processamento em paralelo
  • Rastreabilidade total

06

Comparação: Cenário a Cenário

Modelo Atual (Direto)

  • EME4 fora do ar → erro e perda de dados
  • 500 OPs → uma por uma, lento
  • Formato mudou → altera ambos os sistemas
  • Servidor caiu → tudo para
  • Sem rastreabilidade
  • Sem distribuição de carga

Modelo Novo (Middleware)

  • EME4 fora do ar → fila guarda, entrega depois
  • 500 OPs → todas na fila, paralelo
  • Formato mudou → só altera o Tradutor
  • Servidor caiu → failover automático
  • Rastreabilidade total
  • Load Balancing entre servidores

Resultado com Middleware

Nenhum dado se perde. Se um cai, o outro continua. Cada sistema evolui independentemente. Monitoramento centralizado.

07

Por que confiamos nessas tecnologias?

25M mensagens/seg (NATS)
18K req/seg (Kong/APISIX)
18 MB RAM do NATS
0.2ms latência NATS
280 MB RAM total middleware

NATS

  • Escrito em Go (Google)
  • Usado por Tesla, Mastercard, Walmart
  • CNCF (mesma fundação do Kubernetes)
  • Para nosso volume: 0.000002% da capacidade

Kong / APISIX

  • Construído sobre NGINX (30% da internet)
  • NGINX existe desde 2004 (20+ anos)
  • Usado por Netflix, Samsung, NASA
  • LB + Auth + Failover + Logs grátis

Infraestrutura Total

  • 280 MB RAM (menos que Chrome)
  • < 7% CPU
  • ~1 GB disco
  • Cabe em qualquer VM ou container

Por que IA precisa de Middleware?

  • Agente IA fala com uma porta única em vez de conhecer cada sistema
  • Sem saber a autenticação de cada sistema (sessão, token, API key...)
  • Sem lidar com falhas, timeouts e retentativas
  • Alto volume: centenas de consultas por minuto para montar análises
09

Recomendação

Aprovação da POC do Middleware com escopo reduzido

Escopo da POC

1 cadastro de engenharia (Centro de Trabalho) + 1 fluxo de OP (criar + apontar)

Risco

Baixo — a POC não afeta os sistemas atuais em produção

Infraestrutura

~280 MB RAM em qualquer servidor existente. NATS e Kong/APISIX são gratuitos.

Resultado Esperado

Demonstrar que o Middleware funciona, medir performance real, e ter base concreta para decisão de expansão rumo à IA.

“A POC é um investimento pequeno para uma decisão informada.
Melhor testar antes de comprometer do que comprometer antes de testar.”
